# 運用Runbook (初期版)

このドキュメントは、GitHub SEO Siteの運用において、手動または自動で実行するタスクの手順をまとめたものです。エラー発生時の対応（Troubleshooting）も含みます。

## 1. 定常運用タスク

### 月次レビュー (Analytics & Search Console)

- **頻度**: 月に1回（例：毎月第1月曜日）
- **目的**: サイトへの流入状況、検索順位の確認と、改善ポイントの特定。
- **手順**:
  1. Google Search Consoleを開く。
  2. 検索パフォーマンス（クリック数、表示回数、CTR、平均掲載順位）を前月と比較する。
  3. インデックスカバレッジレポートを確認し、エラー（クロールエラーや「ソフト 404」など）が出ていないかチェックする。
  4. Google Analyticsを開く。
  5. ユーザー獲得元（Organic Searchがどれくらいか）、ページごとの閲覧数、直帰率、滞在時間を確認する。
  6. よく読まれているページやトピックの傾向を掴み、今後のコンテンツ追加（どのジャンルのリポジトリを優先して抽出・生成するか）に活かすよう `targets` リストを調整する。

### Targetsリストの更新 (新規リポジトリ追加)

- **頻度**: 必要に応じて（例：2週間に1回）
- **目的**: 新鮮なコンテンツを追加し、サイトの鮮度を高く保つ。
- **手順**:
  1. `backend/src/main.py` または別途管理する設定ファイル（例：`targets.json`やDB等）を開く。
  2. 新しく話題になっている、またはSEOで狙いたいキーワード・ジャンルのリポジトリを `targets` リストに追加する。
  3. リポジトリに変更をコミット＆プッシュし、GitHub Actions（定期実行もしくは手動実行）をトリガーする。

## 2. トラブルシューティング（エラー発生時の対応）

### 2.1. GitHub Actions のワークフロー失敗ケース一覧

GitHubのActionsタブで赤い×マークが付いた場合、詳細ログから以下の原因を特定し対応します。

#### ① GitHub API Rate Limit エラー
- **ログの特徴**: `403 Forbidden` や `API rate limit exceeded for user ID...`
- **原因**: 1時間あたりのリクエスト上限(PATで通常5,000回)に達した。
- **対応**:
  1. APIリセット時刻まで待機し、Actionsから `workflow_dispatch` で再実行する。
  2. 頻発する場合は1回の処理件数を減らすか、Exponential Backoff(リトライ処理)をコード内に実装する。

#### ② LLM API Failure (OpenAI等)
- **ログの特徴**: `AuthenticationError`, `RateLimitError (429)`, `ServiceUnavailableError (503)`, `ContextWindowExceeded`
- **原因**: APIキー不備、クレジット枯渇、OpenAI側障害、またはREADME文字数超過。
- **対応**:
  1. APIキーとクレジット残高を確認・補充する。
  2. サービス障害時は復旧を待ち再実行する。
  3. ContextWindow超過時は、backend側で対象リポジトリをスキップするか、テキスト切り出し処理を調整する。

#### ③ Quality Check Failure
- **ログの特徴**: スクリプト終了コードが非ゼロで終了、エラーメッセージ（`Validation Error`, `Missing required fields` など）。
- **原因**: LLMが生成したJSONデータがフォーマット違反（タイトル欠損、文字数不足等）を起こした。
- **対応**:
  1. Quality Check スクリプトのログから、どのリポジトリデータの生成に失敗したか特定する。
  2. 問題のあるデータは自動再生成（リトライ）させるか、今回は無視して（削除扱い）パイプラインを継続させるようスクリプト側を堅牢化する。

#### ④ Build Failure (ビルド失敗) / 不要ページの残存
- **ログの特徴**: `npm run build` ステップ内で `Type error: ...` やビルド中断エラー、あるいはビルドは成功したがサイト上に存在しないはずのリポジトリページが表示される。
- **原因**: 
  - LLMが生成したJSONのスキーマが壊れている。
  - フロントエンドのコンポーネント(.tsx)に文法エラーがコミットされた。
  - **Stale Content (古い生成物) の残存**: 今回の生成ではスキップ（除外）されたリポジトリのJSONが `frontend/content/` 内に古いまま残っており、それがビルドに混ざっている。※現在はActions側で毎回 `rm -rf` でクリーンアップする設計にしています。
- **対応**:
  1. ゴミデータ(Stale file)が疑われる場合、`frontend/content/repos` 等の中に古いJSONが残っていないか調査する。
  2. ローカルデバッグの際は、必ず `frontend/content/` 以下の古いデータを手動で削除してから、`backend/output/` の中身をコピー（同期）し直してNext.jsを立ち上げる。
  3. コンポーネント側のエラーの場合はログからエラー箇所を特定し、ローカルで修正してプッシュする。

#### ⑤ Deploy Failure (デプロイ失敗)
- **ログの特徴**: `cloudflare/pages-action` ステップでの認証エラー (`Invalid Token`, `Account ID mismatch`)。
- **原因**: Cloudflare PagesのAPIトークンが無効になった、あるいは設定が誤っている。
- **対応**:
  1. Cloudflareのダッシュボードで新しいAPIトークン（Pages編集権限）を発行する。
  2. GitHub RepositoryのSecrets (`CLOUDFLARE_API_TOKEN`) を更新して再実行する。

### 2.2. 新しいページがCloudflare Pagesに公開されない

- **症状**: Actionsは成功したのにURLアクセスしても古い内容のままになっている。
- **原因の切り分け・対応**:
  - **Cloudflare Pagesのビルド履歴**: Cloudflareのダッシュボードで、最新のデプロイが成功しているか（Stateが `Success` になっているか）確認する。
  - **キャッシュのパージ**: デプロイが成功しているのに古いままなら、ブラウザキャッシュまたはCloudflare CDNのキャッシュが原因。ダッシュボードからキャッシュのパージ(Purge Cache)を行う。

### 2.3. Google Search Console でインデックスエラー警告が来た

- **症状**: 「ページのインデックス登録で問題が検出されました」というメールが届いた。
- **原因の切り分け・対応**:
  - **除外理由の確認**: GSC上でエラーの詳細（例：「クロール済み - インデックス未登録」、「見つかりませんでした (404)」等）を確認する。
  - **対応**:
    - URLがそもそも存在しない（既に削除済み）場合は、リポジトリごとのステータスコードを404ではなく「noindex」を返すようにするか、適切なページへ301リダイレクト設定を入れる。
    - ページ品質が低いと判断された可能性がある場合は、生成されるコンテンツの質を高めるようプロンプト（`llm_generator.py`）や「Quality Check」の基準を見直す。

## 3. リトライ・再実行の基本方針

1. **基本は自動リトライ**: 一過性のエラー（ネットワーク不調や一時的なRate Limit）に対しては、プログラム内部で一定回数（例：3回）リトライさせる組み込みを行う。
2. **手動回復（Manual Intervention）**: 規定回数のリトライでも復旧しない（APIキーの失効や構造的なコードのバグ）場合は、エラーアラート（例：Actionsの失敗通知メール等）を受け取った運用者が手動でログを確認し、原因を修正してから **GitHub Actionsから `workflow_dispatch` (手動トリガー)** により途中から（または最初から）ワークフローを再実行する。
3. **部分再実行**: 既に生成済みのコンテンツは再生成せず（コスト削減のため）、未完了・失敗した分のみを生成する仕組み（差分ビルド）をシステムに持たせる。
